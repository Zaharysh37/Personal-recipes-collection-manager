## Введение

Данный документ описывает план тестирования для приложения ["Personal-recipes-collection-manager"](https://github.com/Zaharysh37/Personal-recipes-collection-manager). Документ предназначен для лиц, выполняющих тестирование данного проекта. Цель тестирования — проверка соответствия реального поведения программы и ее ожидаемого поведения, которое описано в документе с [требованиями (SRS.md)](https://github.com/Zaharysh37/Personal-recipes-collection-manager/blob/master/docs/requirements/SRS.md).

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

*   Авторизация пользователя (регистрация и вход);
*   Просмотр списка рецептов;
*   Поиск по рецептам;
*   Добавление новых рецептов;
*   Редактирование существующих рецептов;
*   Удаление рецептов.

<a name="quality"/>

## Атрибуты качества

1.  **Функциональность**:
    *   **Функциональная полнота**: приложение должно выполнять все заявленные в требованиях функции.
    *   **Функциональная корректность**: приложение должно корректно выполнять все заявленные функции.
2.  **Удобство использования**:
    *   Все функциональные элементы пользовательского интерфейса имеют названия, описывающие действие, которое произойдет при выборе элемента.
    *   Интерфейс интуитивно понятен и соответствует представленным макетам.
3.  **Надежность**:
    *   Приложение должно стабильно работать и корректно обрабатывать ошибки (например, при сбое в работе с локальной базой данных).
4.  **Производительность**:
    *   Приложение должно обеспечивать быстрый отклик при загрузке и поиске рецептов.

<a name="risk"/>

## Риски

К основным рискам можно отнести:
*   Потеря или повреждение данных в локальной базе данных рецептов на устройстве пользователя.
*   Некорректная работа приложения или его отображения на разных версиях ОС Android и устройствах с различными разрешениями экрана.
*   Некорректная обработка пользовательского ввода (например, пустые поля при создании рецепта).

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения. Аспекты, подвергаемые тестированию:
*   Авторизация пользователя;
*   Просмотр списка рецептов и поиск;
*   Просмотр конкретного рецепта;
*   Создание нового рецепта;
*   Редактирование существующего рецепта;
*   Удаление рецепта.

### Авторизация пользователя
Этот вариант использования необходимо протестировать на:
*   Отображение страниц регистрации и входа согласно [макетам](https://github.com/Zaharysh37/Personal-recipes-collection-manager/blob/master/docs/mockups).
*   Успешную регистрацию нового пользователя с корректными данными.
*   Успешный вход существующего пользователя с верными логином и паролем.
*   Обработку ошибок при попытке входа с неверными данными.
*   Обработку ошибок при регистрации с пустыми полями.

### Просмотр списка рецептов и поиск
Этот вариант использования необходимо протестировать на:
*   Отображение главного экрана со списком всех добавленных рецептов.
*   Отображение сообщения о пустом списке, если рецепты еще не добавлены.
*   Корректную работу поиска (поиск существующих и несуществующих рецептов).

### Просмотр конкретного рецепта
Этот вариант использования необходимо протестировать на:
*   Переход на страницу с детальной информацией о рецепте при нажатии на него в списке.
*   Корректное отображение всех данных на странице рецепта: название, описание, ингредиенты, инструкции.

### Создание нового рецепта
Этот вариант использования необходимо протестировать на:
*   Переход на страницу добавления рецепта.
*   Корректное добавление нового рецепта со всеми заполненными полями.
*   Появление нового рецепта в общем списке после его сохранения.
*   Обработку ситуации при попытке сохранить рецепт с незаполненными обязательными полями.

### Редактирование существующего рецепта
Этот вариант использования необходимо протестировать на:
*   Вызов диалогового окна для редактирования рецепта (длительное нажатие на элемент списка).
*   Открытие экрана редактирования с уже заполненными данными рецепта.
*   Корректное сохранение изменений и их отображение в списке и на странице рецепта.

### Удаление рецепта
Этот вариант использования необходимо протестировать на:
*   Вызов диалогового окна для удаления рецепта (длительное нажатие на элемент списка).
*   Успешное удаление рецепта из списка после подтверждения в диалоговом окне.
*   Отмену удаления рецепта.

### Нефункциональные требования:
*   Проверить, что все элементы интерфейса (кнопки, поля ввода, заголовки) имеют понятные названия.
*   Оценить скорость загрузки списка рецептов и отклика интерфейса на действия пользователя.

При тестировании будет использован ручной подход (manual testing). Каждый аспект будет проверяться вручную согласно описанным выше сценариям.

Данный тестовый план позволяет провести комплексную проверку основного функционала приложения "Personal-recipes-collection-manager". Успешное прохождение всех тестов, описанных в плане, позволит с высокой долей уверенности утверждать, что приложение работает корректно и соответствует заявленным требованиям.
